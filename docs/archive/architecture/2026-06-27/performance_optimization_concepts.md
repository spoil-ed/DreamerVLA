# 性能优化知识点与差异点

> [!abstract] 这是什么
> 这是性能优化的**知识篇**(逐项落到本仓哪一行/哪个 commit,见 [`HISTORY.md`](../HISTORY.md) 已合并项与 [`TODO.md`](../superpowers/TODO.md) 待办项)。本篇只讲**原理和差异点**——每个优化技巧背后的机制、朴素写法为什么慢、优化写法**差在哪一步**、以及以后怎么一眼认出同类问题。
>
> 本篇刻意**不下到具体代码**（不写文件行号、不贴代码），适合用来建立判断力，而不是照着改。需要落地位置时再回到审查篇。

> [!tip] 怎么用这份文档
> 每个知识点都按同一套讲：**原理 → 差异点（朴素↔优化）→ 识别信号**。
> "差异点"是核心：它指出朴素与优化**到底差在哪一个动作**——往往不是"换个算法"，而是"换个时机 / 换个布局 / 换个粒度"。

## 0. 一张总表（先看这个）

| 知识点 | 朴素的"信号词" | 优化的"动作" | 差异本质 |
|--------|----------------|--------------|----------|
| GPU 同步点 | 循环里 `.item()/.cpu()` | 累加在 GPU，循环外取一次 | 同步的**时机/次数** |
| 推理上下文 | `no_grad` | `inference_mode` | 是否还维护 autograd 版本计数 |
| 混合精度 | 整模型静态 `.to(bf16)` | 参数留 fp32 + `autocast` | fp32 **master 权重**在不在 |
| 升精度 | 整张大张量 `.float()` | 只对归约结果升精度 | 升精度的**范围** |
| 峰值显存 | 整条轨迹一次 backward | 微批 / 梯度累积 | 同时存活的**计算图大小** |
| 自回归 | 每步重算整段历史 | KV-cache 增量 | 复杂度 O(W²)→O(W) |
| 注意力 | 手写 einsum 物化 score | Flash / SDPA | 是否**物化**完整注意力矩阵 |
| 不变量 | 每步重建 mask/位置编码/prompt | 缓存 / 预建 buffer | 重算 vs 复用**常量** |
| 数据布局 | list-of-dict 每次重组 | 每字段连续数组 | AoS vs SoA |
| 读取粒度 | 每次读整段再切 | 按需切片读 | 读取的**数据量** |
| DataLoader | `pin_memory=false`、阻塞 `.to()` | pin + `non_blocking` + 预取 | CPU 取数与 GPU 算**重不重叠** |
| 多张量算子 | Python 逐参数循环 | `_foreach_` 融合 | kernel 启动**次数** |
| 逐元素循环 | for + `.item()` 填值 | `scatter` / fancy-index | 在 Python 还是在 kernel 里循环 |
| 计算-IO 重叠 | 采集/落盘内联串行 | 双缓冲 / 后台线程 | CPU 与 GPU 谁在等谁 |
| 分布式通信 | 每个指标一次 all_reduce | 堆叠后一次 all_reduce | 集合操作的**次数** |
| 冗余拷贝 | `.detach().cpu().clone()` | 去掉多余 `.clone()` | 复制了**几份** |
| 诊断日志 | 每步无条件算指标 | 按 cadence / debug flag | 为日志**付几次**算力 |

## 1. GPU 执行模型：一切同步问题的根

> [!note] 核心心智模型
> GPU 是**异步**的。CPU 发出的 kernel（矩阵乘、卷积…）只是被排进 GPU 队列，CPU 不等它算完就继续往下跑。只有当你**要把结果读回 CPU** 时，CPU 才被迫停下来等队列清空——这就是"同步点"。

### 1.1 同步点（`.item()/.cpu()/.numpy()/.tolist()`）

- **原理**：训练理想状态是 CPU 一直在前面"喂"kernel、GPU 一直满负荷。任何把张量值读回 CPU 的操作都会插入一道屏障：CPU 必须等到该值真正算出来。一次同步可能就是几百微秒到几毫秒。
- **差异点**：
  - 朴素：在**每步 / 每 minibatch 循环内部**就把标量读回 CPU（比如累加 loss、打印、写日志）。于是每次迭代都被强制同步一次，GPU 频繁"等 CPU 等它"，流水线被切碎。
  - 优化：把要累加的标量**留在 GPU 上做加法**，整个循环跑完**只在最后读回一次**。需要的多个标量先堆成一个张量、一次性搬回。
  - **本质差异不是"少算了什么"，而是"把 N 次同步合并成 1 次"**——计算量没变，但 GPU 不再被反复打断。
- **识别信号**：循环体里出现 `.item()`、`.cpu()`、`.numpy()`、`.tolist()`、`print(张量)`；尤其紧跟在 `backward()` 后面的 `.item()`，它会一直阻塞到反向传播算完。

### 1.2 `no_grad` 与 `inference_mode` 的差异

- **原理**：推理时不需要梯度。`no_grad` 不建反向图，但 PyTorch 仍为每个张量维护"版本计数"等 autograd 簿记；`inference_mode` 更彻底，连这些簿记都关掉。
- **差异点**：朴素用 `no_grad`（已经不错）；优化在纯推理 / 冻结模型前向用 `inference_mode`，省掉那层薄记账开销与少量内存。**差异很小但免费**——前提是这些张量之后不会再进入需要梯度的计算。
- **识别信号**：rollout、评估、冻结编码器前向里用的是 `no_grad`。

## 2. 显存与数值精度

### 2.1 混合精度：`autocast` vs 静态 cast

> [!warning] 这是最容易"自以为做了优化其实埋了坑"的地方

- **原理**：bf16/fp16 算得快、省显存，但**优化器更新需要 fp32 精度**（Adam 的动量、极小的权重增量在 fp16 下会被舍入吃掉）。标准做法是"**计算用低精度、权重存高精度**"。
- **差异点**：
  - 朴素：把整个模型 `.to(bf16)` 一刀切。前向是快了，但**参数本身变成了 bf16**，于是优化器的 master 权重也没了——小步更新被舍入，收敛变差；fp16 还可能溢出却没有保护。
  - 优化：**参数始终留 fp32**，只在前向/反向外面套一层 `autocast`，让框架自动决定哪些算子用 bf16、哪些保留 fp32；fp16 路径再配 `GradScaler` 防下溢。
  - **差异本质：fp32 master 权重在不在**。静态 cast 把"算"和"存"用了同一种低精度，autocast 把两者分开。
- **识别信号**：代码里出现把整个模型 `.to(dtype=bf16/fp16)`，而**找不到** `autocast` / `GradScaler`。

### 2.2 升精度的"范围"

- **原理**：为了数值稳定（如算 loss、归一化、cosine），把张量升回 fp32 是对的——但**只需要对参与归约的那一小步升精度**，不需要把整张大特征图都升。
- **差异点**：朴素把 `[B,T,N,D]` 这种大张量整体 `.float()` 再算 MSE，于是显存里凭空多出两份 fp32 大副本（预测 + 目标）。优化只对**最终的求和/求均值**用 fp32 累加，逐元素差仍在低精度做。**差异点是升精度发生在归约前还是归约后**。
- **识别信号**：对预测张量和目标张量**同时** `.float()`，紧接着才做 `mse`/`normalize`。

### 2.3 峰值显存：微批 / 梯度累积

- **原理**：反向传播需要保留前向的**激活值**（计算图）。同时存活的图越大，峰值显存越高。显存是被**峰值**而不是平均值决定的——一次 OOM 整个训练就崩。
- **差异点**：
  - 朴素：把一整条 imagined 轨迹（或一个大 batch）展开成一个巨大的计算图，**一次 `backward()`**。峰值 = 整条轨迹的全部激活。
  - 优化：把它切成小片，**一片一片前向+反向、即时释放每片的图**，梯度累加；或梯度累积（多个小 batch 累积梯度再 step 一次）。计算总量不变，但**任一时刻只有一小片的图存活**，峰值显存大幅下降。
  - **差异本质：同时存活的计算图大小**。这正是本仓 `outcome` 路径相对 `dense` 路径已经做对的事。
- **识别信号**：有个 `gradient_accumulate_every` 之类的旋钮但循环里没真用；或一条很长的序列展开后只有一次 backward；训练在大 batch 下 OOM。

### 2.4 诊断带来的"隐形反向"

- **原理**：很多梯度诊断（各 loss 分量的梯度范数、夹角）需要**额外的反向传播**。一次额外 `autograd.grad(retain_graph=True)` 就是一次完整反向，而 `retain_graph` 会让整张图在多次反向间一直占着显存。
- **差异点**：朴素**每一步**都为诊断跑好几次额外反向，且全程 `retain_graph`；优化把这些诊断**默认关掉或按 N 步采样一次**，主路径只留一次真正用于更新的 `backward()`。**差异点是"为了看指标，每步多付了几次反向"**。
- **识别信号**：训练步里出现多次 `autograd.grad(...)` 且结果只用于打日志；`retain_graph=True`。

## 3. 注意力与自回归

### 3.1 KV-cache：自回归的 O(W²)→O(W)

> [!important] 这是世界模型 / VLA 解码最大的一类浪费

- **原理**：自回归（imagination、token 解码）每生成一步，新 token 只需要和**历史的 K/V**做注意力。历史的 K/V 在前面的步里**已经算过了**。
- **差异点**：
  - 朴素：每步都把"整段历史 + 新 token"重新喂进 Transformer，从头算一遍全部注意力，最后只取最后一个位置的输出——前面 W-1 步的 K/V 被反复重算。单条长度 W 的 rollout 总开销是 **O(W²)**。
  - 优化：把每层每步算出的 K/V **缓存**起来，下一步只为**新 token**算 Q 并与缓存的 K/V 做注意力。总开销降到 **O(W)**。
  - **差异本质：历史 K/V 是重算还是复用**。
- **识别信号**：单步推理函数内部对"整段历史"跑完整 attention，然后 `[:, -1:]` 只取末位；窗口/历史用 `cat` 每步往后滚。
- **注意**：当历史窗口本身很短（如 `num_hist` 只有 2~3）时，KV-cache 收益有限，这时更值钱的是**别重复编码同一帧**、**别在循环里反复 `cat` 重建整段张量**。

### 3.2 Flash / SDPA vs 手写注意力

- **原理**：注意力的中间产物——`[Tq, Tk, B, Head]` 的 score 矩阵——是显存大户。手写实现会把它**完整物化**在显存里再 softmax；Flash-Attention 类 kernel 用分块流式计算，**从不把完整 score 矩阵落到显存**。
- **差异点**：朴素用 einsum 显式算出整张 score 矩阵、`masked_fill`、softmax，序列一长就吃大量显存和带宽，还常伴随 `.float()` 再 `type_as` 的精度往返；优化调用框架的 `scaled_dot_product_attention`，让它自动选 Flash / memory-efficient kernel。**差异点是完整注意力矩阵到底有没有在显存里出现过**。
- **识别信号**：注意力是手写 `einsum`/`bmm` + `softmax`，没走 `scaled_dot_product_attention`；序列越长越容易 OOM。

### 3.3 缓存"不变量"

- **原理**：因果掩码、正弦位置编码、固定 prompt 的 token——这些只依赖序列长度或文本内容，在整段 rollout / 整个 epoch 里**根本不变**。
- **差异点**：朴素在**每一步 / 每个 forward** 都重新构造它们（重新 `arange`、重新 `tril` 出一个大矩阵、重新 tokenize 同一句话）。优化把它们按 `(长度, device, dtype)` **缓存**，或预建到最大长度的 buffer 再切片复用；不变的 prompt token **只 tokenize 一次**。**差异点是"每步重建一个常量" vs "建一次复用"**。
- **识别信号**：每步/每 forward 里出现构造 mask、位置编码、对同一句指令反复 tokenize。

## 4. 数据布局与 IO

### 4.1 AoS vs SoA：list-of-dict 的代价

> [!note] 一个"看起来很自然、其实很贵"的写法

- **原理**：存一批样本有两种布局。**AoS**（Array of Structs）= 一个列表，每个元素是一个 dict（一步的所有字段）；**SoA**（Struct of Arrays）= 每个字段一整块连续数组。GPU/numpy 喜欢**连续大块**。
- **差异点**：
  - 朴素（AoS）：把每条轨迹存成"每步一个 dict 的列表"。每次取一个 batch，都要用 Python 嵌套循环把成百上千个小数组**重新拼**成大张量（`np.stack`），对高维向量（如几十万维的 embedding）尤其昂贵——每次采样都是一次大规模内存搬运 + Python 开销。
  - 优化（SoA）：**入库时**就把每条轨迹按字段存成连续数组；采样时只做**切片**（几乎零拷贝的视图）再一次性堆叠。
  - **差异本质：每次采样是"重新组装" vs "切片视图"**。
- **识别信号**：replay / dataset 里出现 `[[step[k] for step in window] for window in windows]` 这类嵌套推导，或双重 `for` 循环逐元素填张量。

### 4.2 读取粒度：按需切片 vs 整段读

- **原理**：HDF5 / 内存映射支持**只读你要的那几行**。但很容易写成"把整段读进来再切"。
- **差异点**：朴素每次 `__getitem__` 把整条轨迹的某个字段（如全部 actions）读进内存，然后只用其中几行；同一条轨迹有 T 个样本，就读了 T 次整段 → 总读取量 **O(T²)**。优化直接切片读 `[start:start+需要的长度]`，总读取量降到 **O(T·需要长度)**。**差异点是 read 的边界**。
- **识别信号**：`__getitem__` 里把整个字段 `np.asarray(整段)` 后再 `[索引]`。

### 4.3 per-worker 句柄缓存与构造期成本

- **原理**：打开文件、反序列化 pickle 是有固定开销的。多 worker 的 DataLoader 里，每个 worker 应该**只打开一次文件**、对热数据**做缓存**。
- **差异点**：朴素每次取数都重新 `open` / `pickle.load`，相邻的滑动窗口高度重叠却各读各的；或在 `__init__` 里把整个数据集解一遍 pickle 只为建索引。优化用**每 worker 一个文件句柄缓存**、对重叠帧加 LRU 缓存、构造期优先用轻量 manifest 而不是反序列化整条记录。**差异点是"重复打开/解析" vs "打开一次、复用"**。
- **识别信号**：`__getitem__` 里 `open(...)` / `pickle.load(...)`；`__init__` 里遍历整库做反序列化。

### 4.4 DataLoader 流水线：pin_memory + non_blocking + 预取

- **原理**：理想流水线是"worker 在 CPU 上准备下一批" 与 "GPU 在算当前批" **同时进行**。要让 H2D 拷贝能和计算重叠，需要两件事配齐：host 端用**锁页内存**（`pin_memory`），拷贝用**异步**（`non_blocking=True`）。两者缺一，拷贝就退化成阻塞。
- **差异点**：
  - 朴素：`pin_memory=false` + 普通 `.to(device)`（同步）+ `prefetch_factor=1`（几乎不预读）。于是每批数据"现取现搬"，GPU 取数时干等。
  - 优化：`pin_memory=true` + `.to(device, non_blocking=True)` + 适当增大 `prefetch_factor`、`persistent_workers=true`。取数与计算重叠起来。
  - **差异本质：CPU 取数和 GPU 计算是串行还是重叠**。注意 `non_blocking` 只有在**锁页内存**下才真正异步——这就是为什么两者必须成对出现。
- **识别信号**：config 里 `pin_memory: false`；device move 没有 `non_blocking`；`prefetch_factor` 很小。

## 5. 向量化：把循环交给 kernel

### 5.1 融合多张量算子（`_foreach_`）

- **原理**：每次启动一个 GPU kernel 有固定的"发射开销"。对一个有几千个参数张量的模型，逐个张量做两步操作 = 上万次微型 kernel 发射，**瓶颈在发射而不在计算**。
- **差异点**：朴素用 Python `for` 遍历每个参数、各发两个 kernel（如 EMA 的 `mul_` 再 `add_`）。优化用 `torch._foreach_*` 把"对一组张量做同样操作"融合成**一次** multi-tensor kernel。**差异点是 kernel 发射次数：O(参数数) → O(1)**。
- **识别信号**：`for ... in model.parameters()` 里逐个做 `mul_/add_/copy_`。

### 5.2 批一次前向 vs 逐样本/逐窗口/逐帧循环

- **原理**：一批互相独立的小输入，堆成一个大 batch 一次前向，远比 Python 循环里逐个前向快（同样是省 kernel 发射 + 打满并行）。
- **差异点**：朴素用 Python 循环"逐窗口跑一次分类器"、"逐帧调一次世界模型头"、"逐样本过一次图像 processor"。优化把它们 reshape/`unfold` 成一个大 batch、**一次前向**算完再 reshape 回去。**差异点是循环在 Python 里还是被折叠进 batch 维**。
- **识别信号**：`for` 循环体里调用模型 / processor，然后 `stack` 结果。
- **权衡**：批化有时会让"提前 break"之类的早停失效，但通常吞吐净赚。

### 5.3 `scatter` / fancy-index vs 逐元素 `.item()` 填值

- **原理**：构造稀疏张量（比如把每条样本在某个位置置 1）是一个纯索引赋值，可一次性向量化完成。
- **差异点**：朴素 `for i in range(B): if cond[i].item(): out[i, idx[i].item()] = 1`——每次迭代两次 `.item()`（两次同步）。优化用 `out[arange(B), idx] = vals` 或 `scatter_`，**零循环、零同步**。**差异点是"在 Python 里逐个写" vs "一条索引赋值"**。
- **识别信号**：`for` + `.item()` 用来决定往张量哪里写值。

## 6. 异步与并发：让谁都别干等

### 6.1 计算-IO 重叠（双缓冲 / 后台落盘）

- **原理**：仿真环境 `step` 在 CPU 上、策略前向在 GPU 上。如果严格串行（先 step 再 forward 再 step…），GPU 在 step 时空转、CPU 在 forward 时空转，吞吐被两者**之和**限制，而不是**较大者**。
- **差异点**：朴素单环境严格串行；优化要么**向量化多环境**（一次前向喂多个 env），要么**双缓冲**（在算当前批前向的同时，让下一批 env 去 step）。同理，视频编码、写盘这类 IO 朴素内联在采集线程里阻塞，优化丢给**后台线程**消费。**差异本质：吞吐 = max(CPU, GPU) 而不是 CPU+GPU**。
- **识别信号**：采集循环是"单 env：取观测→前向→step"严格串行；编码/落盘函数直接在主循环里调用。

### 6.2 Ray：fire-and-forget 与批量取结果

- **原理**：`ray.get` 是阻塞的——它会等远端 RPC 回来。把它放进循环就把本可并行的远程调用串行化了。
- **差异点**：
  - 朴素：循环里每次 `ray.get(远程调用)`，下一次必须等上一次 RPC 往返完成；或在每个 episode 边界阻塞等 replay 写完才继续采集。
  - 优化：**先把所有远程调用的 ref 发出去，再一次 `ray.get` 全部**；写 replay 这类"发了不必等"的操作 fire-and-forget，只在必要时（有界 backpressure）回收。
  - **差异本质：RPC 延迟是串行累加还是相互重叠**。
- **识别信号**：`for ...: ray.get(...)`；每个 episode 结束阻塞等远程 actor。

### 6.3 冗余拷贝：多余的 `.clone()` / 多次序列化

- **原理**：要分清一个操作**已经产生了新对象**还是只是个视图。对 GPU 张量，`.detach().cpu()` **已经是一块全新的 CPU 内存**（不是 GPU 内存的视图），后面再 `.clone()` 就是无意义地再复制一份整模型。
- **差异点**：朴素 `.detach().cpu().clone()`，对几十亿参数的模型就是多出一份完整 CPU 拷贝；若之后还要 `ray.put` 序列化，又是一份。优化去掉多余的 `.clone()`。**差异点是"复制了几份"**——前提是这份数据在序列化前不会被原地改写。
- **识别信号**：`.detach().cpu().clone()` 这种链；为同一份权重在传输前反复拷贝。

## 7. 工程纪律（最便宜、最该先做）

### 7.1 诊断 / 日志按 cadence 收口

- **原理**：日志只在你**真要看的那一步**才需要。但很容易写成"每步都把所有指标算好、搬回 CPU、reduce 好，最后才判断这步要不要记"。
- **差异点**：朴素**先付费再判断**——每步无条件物化几十个指标（每个一次同步）、甚至每步 shell 出去调 `nvidia-smi`，然后才看 `% log_every`。优化把"算指标 + 搬回 + reduce"整块**放进 `if 该记录` 分支**，非记录步一次同步都不做。**差异本质：为日志付的代价是 per-step 还是 per-log**。
- **识别信号**：指标字典的构造在 `if step % log_every == 0` 判断**之前**；子进程调用（nvidia-smi）按迭代而非按 epoch。

### 7.2 集合通信合并

- **原理**：分布式里每次 `all_reduce` 都有固定的 RPC/同步延迟。N 个小标量各 reduce 一次 = N 次延迟。
- **差异点**：朴素逐个指标 `all_reduce`（几十个 train 指标 = 几十次小集合操作 + 几十次同步）。优化把它们**堆成一个张量、一次 `all_reduce`、再切回**。**差异点是集合操作的次数**。
- **识别信号**：`for k in metrics: all_reduce(metrics[k])`。

## 8. 把这些串起来：一个判断清单

> [!check] 以后看到任何热路径代码，按这个顺序自问
> 1. **同步**：这个循环里有没有 `.item()/.cpu()`？能不能挪到循环外？（§1.1）
> 2. **诊断**：这步算的东西是不是只为打日志？能不能 gate 掉？（§2.4, §7.1）
> 3. **粒度**：这是不是把一整条轨迹/一个大 batch 一次性展开了？能不能微批？（§2.3）
> 4. **重算**：有没有把"常量"（mask、位置编码、prompt、历史 K/V、冻结编码器输出）每步重建？（§3.1, §3.3）
> 5. **布局**：数据是 list-of-dict 吗？读取是不是读了整段只用一点？（§4.1, §4.2）
> 6. **流水线**：CPU 取数 / 仿真 与 GPU 计算重叠了吗？（§4.4, §6.1）
> 7. **循环**：这个 Python 循环能不能折叠成一次 batch 前向 / 一个 `scatter` / 一个 `_foreach_`？（§5）
> 8. **精度**：是静态 cast 还是 autocast？有没有整张大张量无谓升精度？（§2.1, §2.2）
> 9. **拷贝**：有没有多余的 `.clone()` / 多次序列化？（§6.3）

> [!quote] 一句话
> 这些优化里**几乎没有"换更聪明的算法"**——它们换的是**时机**（同步挪到循环外、日志按需）、**布局**（SoA、连续数组）、**粒度**（微批、批前向）、**复用**（缓存常量与 K/V）、**重叠**（计算与 IO 并行）。识别它们靠的不是背算法，而是认出上面那些"信号词"。具体落到本仓哪一行，见 [`HISTORY.md`](../HISTORY.md)(已合并项)与 [`TODO.md`](../superpowers/TODO.md)(待办项)。
